  GDPR PRIVACY APP — FULL IMPLEMENTATION PLAN
============================================================

PROJECT OVERVIEW
----------------
A full-stack GDPR-compliant privacy management application.
Users can view, edit, export, and delete their personal data.
Admins can monitor GDPR requests and audit logs.

Stack:
  Backend  — FastAPI + PostgreSQL + SQLAlchemy + JWT
  Frontend — React + TypeScript + Tailwind CSS + Shadcn/ui
  Charts   — Recharts
  Deploy   — Docker Compose (3 services)

      updated_at, ip_address

  - AuditLog model:
      id, user_id (FK), action (ENUM), timestamp, ip_address, details
      ENUM values: ACCESS, DELETE, EXPORT, RECTIFY, CONSENT_CHANGE,
                   LOGIN, REGISTER, ANONYMISE

1.5  Alembic Migrations

  - Create initial migration for all tables

PHASE 2 — AUTH ENDPOINTS (Day 2)
============================================================

2.1  Auth Schemas (app/auth/schemas.py)
  - UserCreate: email, password, first_name, last_name, phone, address
  - UserLogin: email, password
  - Token: access_token, token_type
  - UserResponse: id, email, first_name, last_name, is_admin

2.2  Auth Utilities (app/auth/utils.py)
  - hash_password(plain) → bcrypt hash
  - verify_password(plain, hashed) → bool
  - create_access_token(data, expires_delta) → JWT string
  - get_current_user(token) → User — FastAPI dependency

2.3  Auth Router (app/auth/router.py)
  POST /auth/register
    - Validate email uniqueness
    - Hash password, create User record
    - Create default consent records (all False except essential)
    - Log REGISTER to AuditLog
    - Return JWT

  POST /auth/login
    - Verify email + password
    - Log LOGIN to AuditLog with IP
    - Return JWT

  GET /auth/me
    - Protected — return current user profile

PHASE 3 — GDPR ENDPOINTS (Days 3–4)
============================================================

3.1  GDPR Schemas (app/gdpr/schemas.py)
  - ConsentUpdate: consent_type, granted
  - UserUpdate: first_name, last_name, phone, address
  - AuditLogResponse: action, timestamp, ip_address, details
  - ExportData: full user data bundle

3.2  GDPR Router (app/gdpr/router.py)

  GET /me/data
    - Return full user profile (Article 15 — Right of Access)
    - Log ACCESS to AuditLog

  PUT /me/data
    - Update user fields (Article 16 — Right to Rectification)
    - Log RECTIFY to AuditLog

  DELETE /me/data
    - GDPR "Right to be Forgotten" (Article 17)
    - Strategy: Anonymise PII rather than hard delete
        email → anon_{uuid}@deleted.gdpr
        first_name, last_name, phone, address → "[REDACTED]"
        is_active = False
        anonymised_at = now()
    - Revoke all consents
    - Log ANONYMISE to AuditLog

  GET /me/export
    - Bundle all user data (profile + consents + audit logs)
    - Return as JSON response (Article 20 — Data Portability)
    - Log EXPORT to AuditLog

  GET /me/consents
    - Return all consent records for current user

  PUT /me/consents
    - Update a single consent (toggle on/off)
    - Log CONSENT_CHANGE to AuditLog

  GET /me/audit-logs
    - Return paginated audit log for current user
    - Query params: action_filter, page, page_size

3.3  Admin Router (app/admin/router.py)
  (Requires is_admin=True on JWT)

  GET /admin/users
    - Paginated list of all users (anonymised fields hidden)

  GET /admin/audit-logs
    - All audit logs across all users
    - Filter by action, user_id, date range

  GET /admin/stats
    - Total users count
    - Deletion requests (ANONYMISE count)
    - Export requests count
    - Consent withdrawals count
    - Daily GDPR request counts (last 30 days) — for chart

3.4  CORS + Middleware (app/main.py)
  - CORSMiddleware: allow_origins=["http://localhost:3000"]
  - TrustedHostMiddleware for production
  - Custom middleware to capture real IP for audit logging

  - Request interceptor: attach Bearer token from memory

      auth: register(), login(), getMe()
      gdpr: getMyData(), updateMyData(), deleteMyData()
            exportMyData(), getConsents(), updateConsent()

  - Admin routes:   /admin (requires isAdmin flag)
  - Redirect unauthenticated users to /login

PHASE 6 — DOCKER + DEVOPS (Day 9)
============================================================

6.1  Backend Dockerfile
  FROM python:3.12-slim
  WORKDIR /app
  COPY requirements.txt .
  RUN pip install -r requirements.txt
  COPY . .
  CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]

6.2  Frontend Dockerfile
  FROM node:20-alpine AS builder
  WORKDIR /app
  COPY package*.json .
  RUN npm ci
  COPY . .
  RUN npm run build

  FROM nginx:alpine
  COPY --from=builder /app/dist /usr/share/nginx/html
  EXPOSE 3000

6.3  docker-compose.yml
  services:
    db:
      image: postgres:16
      environment: POSTGRES_DB, POSTGRES_USER, POSTGRES_PASSWORD
      volumes: postgres_data:/var/lib/postgresql/data
      healthcheck: pg_isready

    backend:
      build: ./backend
      depends_on: db (service_healthy)
      environment: DATABASE_URL, SECRET_KEY
      ports: "8000:8000"

    frontend:
      build: ./frontend
      depends_on: backend
      ports: "3000:80"

6.4  Environment Files
  backend/.env:
    DATABASE_URL=postgresql+asyncpg://...
    SECRET_KEY=<random 64 char hex>
    ALGORITHM=HS256
    ACCESS_TOKEN_EXPIRE_MINUTES=60

  .env.example — committed to repo (no real secrets)

KEY ARCHITECTURAL DECISIONS
============================================================

JWT in Memory (not localStorage)
  Reason: localStorage is vulnerable to XSS attacks. Storing JWT
  in React state means it is cleared on page refresh — intentional
  for GDPR apps where session persistence is a liability.

Anonymisation over Hard Delete
  Reason: Hard deleting rows breaks referential integrity and audit
  trails. GDPR's "Right to Erasure" requires that PII is no longer
  identifiable — anonymisation achieves this while preserving audit
  log integrity for legal compliance.

Async SQLAlchemy
  Reason: FastAPI is async-native. Using asyncpg + async SQLAlchemy
  maximises throughput for concurrent GDPR request handling.

UUID Primary Keys
  Reason: Sequential integer IDs leak user count and are predictable.
  UUIDs are safer for a privacy-focused application.

Consent as Separate Table
  Reason: Consent history must be auditable under GDPR Article 7.
  A separate table with timestamps allows you to prove what was
  consented to and when.
